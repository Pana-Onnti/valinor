⚡ D4C Internal — v1.0 Linear-First

Agent Infrastructure

Guía para el equipo Delta 4C. Cómo trabajamos con Claude Code, integrado con Linear como source of truth y GitHub para código. Adaptado del sistema StockIA, evolucionado para un equipo de 3.

01 — La tesis

¿Por qué no copiamos StockIA tal cual?

StockIA puso todo su project management en archivos markdown (.stockia/ROADMAP.md, SESSION_LOG.md, tasks/) porque su agente no podía hablar con APIs externas. Cada estado vivía en el filesystem.

Delta 4C tiene Linear via MCP. El agente puede leer issues, crear documentos, actualizar estado — directamente. Esto cambia la arquitectura de raíz:

📋

Linear = PM source of truth

Roadmap, backlog, issues, ciclos, decision log, session tracking — todo vive en Linear. No duplicamos en markdown.

🔀

GitHub = código source of truth

Branches, commits, PRs. Cada commit referencia un issue de Linear. El código nunca se trackea fuera de git.

📂

Filesystem = agente context only

CLAUDE.md, subagentes, commands, y docs de arquitectura que el agente necesita sin hacer API calls.

🔗

Loop cerrado

Linear → branch → code → commit (Refs: VAL-XX) → push → update Linear. Trazabilidad completa sin holes.

Principio fundacional: Si algo necesita ser leído por humanos Y agentes → Linear. Si algo necesita ser leído por el agente durante coding sin latencia → filesystem. Si algo necesita versionarse → git. Nunca duplicar entre capas.
02 — Mapa de source of truth

Dónde vive cada cosa

Info Dónde Por qué ahí

Issues / Tareas Linear Linear Issues por team (VAL, NAR, GRO) Ciclos, prioridades, assignees — todo nativo de Linear.
Decision Log Linear Linear Doc: "Decision Log" Registro compartido. El agente appenda decisiones vía MCP.
Session Log Linear Linear Doc (nuevo): "Session Log — Dev" Reemplaza .stockia/SESSION_LOG.md. Persiste entre repos.

Código fuente Git src/ en repo Branches por issue. Conventional commits.
Playbooks, KO Design, etc. Linear Linear Docs en Knowledge Base Referencia de equipo. El agente los lee cuando necesita contexto de negocio.

04 — CLAUDE.md

El router — <2KB que se lee siempre

Router

Template para CLAUDE.md

# Delta 4C — Valinor

## Reglas no negociables
1. Conventional commits: `tipo(scope): desc` + `Refs: VAL-XX` obligatorio
2. Domain nunca importa de Infrastructure (hexagonal)
3. Tests antes de commit: `pytest tests/ -v` debe pasar
4. Todo issue se trackea en Linear — no crear tareas sueltas
5. Un branch = un issue de Linear

## Build & Test
```
docker compose up -d
pytest tests/ -v
```

## Workflow: Linear → Git → Code → Linear
PICK issue → BRANCH (Linear name) → CODE → COMMIT → UPDATE Linear

## Contexto on-demand
- Arquitectura: `docs/ARCHITECTURE.md`
- Guía de dev: `docs/DEVELOPER_GUIDE.md`
- Domain model: `docs/DOMAIN_MODEL.md`
- Issues activos: `Linear:list_issues` (MCP)
- Decisiones: Linear Doc "Decision Log" (MCP)

## Agentes: `.claude/agents/`
## Commands: `.claude/commands/` (start-session, end-session, etc.)
## Skills: `.claude/skills/` (d4c-linear-workflow, d4c-brand)
¿Por qué <2KB? Claude Code lee CLAUDE.md en CADA interacción. Cada línea compite por atención. El router solo tiene reglas + punteros. La información detallada se carga on-demand desde docs/ o desde Linear vía MCP.
05 — Subagentes

6 especialistas para Delta 4C

Adaptados del sistema StockIA al contexto D4C. Cada uno tiene su propio contexto, tools permitidas, y modelo óptimo.

Agente Rol en D4C Tools Model
swarm-architect Diseño del pipeline multi-agente. Valida que el flujo Cartógrafo→Analista→Narrador se respete. Revisa harnesses y tools de cada Vala. Read, Grep, Glob Sonnet
backend-dev Implementa features en FastAPI + Celery + Redis + PostgreSQL. Sigue DEVELOPER_GUIDE.md para el flujo Domain→App→Infra. Read, Write, Edit, Bash Sonnet
test-writer Tests unitarios, de integración, y E2E. Cobertura de los agentes del swarm, API endpoints, y data pipeline. Read, Write, Bash Haiku
infra-ops Docker Compose, Prometheus, deploy via SSH tunneling, ZeroTrust. Monitoring y alertas. Caddy reverse proxy. Read, Write, Edit, Bash Haiku
agent-engineer Escribe y refina harnesses (system prompts), define tools para cada Vala, optimiza token usage, calibra Haiku vs Sonnet. Read, Write, Edit, Bash Sonnet
pm-linear Lee y actualiza Linear: issues, session log, decision log. Genera dashboards de progreso. Sincroniza estado. Read, Write, Glob, Linear MCP Haiku
Diferencia clave vs StockIA: Tenemos agent-engineer (escribe harnesses para los Valar) y pm-linear (opera sobre Linear via MCP). StockIA tenía security-reviewer y data-engineer — los nuestros son infra-ops y swarm-architect porque nuestro dominio es agentic AI, no fintech.
06 — Commands

Slash commands con Linear integration

La diferencia principal: los commands de StockIA leían/escribían markdown. Los nuestros leen/escriben Linear vía MCP.

Comando Qué hace Source
/project:start-session Lee issues "In Progress" de Linear + active-plan.md local. Presenta dónde quedamos y qué sigue. Si hay un plan activo, lo retoma. Linear + FS
/project:end-session Guarda resumen en Linear Doc "Session Log". Actualiza estado de issues trabajados. Actualiza active-plan.md local. Linear + FS
/project:status Dashboard: issues del cycle actual, % completado, blockers, próximas prioridades. Todo desde Linear. Linear
/project:plan-task [desc] Investiga codebase, define scope, acceptance criteria. Crea issue en Linear con label, priority, estimate. NO crea archivos markdown. Linear + FS
/project:implement-feature [issue] Lee issue de Linear, crea branch, ejecuta Domain→App→Infra. Commits con Refs: VAL-XX. Al terminar, actualiza Linear. Linear + Git
/project:fix-bug [issue] Lee issue, reproduce, diagnostica, fix, test, commit. Actualiza Linear al cerrar. Linear + Git
/project:run-tests Ejecuta pytest tests/ -v. Reporta resultados. Si falla, sugiere fix. FS
/project:review-code Delega a swarm-architect + infra-ops para revisión dual: arquitectura + seguridad/infra. FS
07 — Session lifecycle

El "git pull/push" cognitivo

/project:start-session — Paso a paso

Start

Lo que hace el agente al arrancar

# 1. Leer estado de Linear
→ Linear:list_issues (assignee=me, state="In Progress")
→ Linear:list_issues (assignee=me, state="Todo", cycle=current)

# 2. Leer plan táctico local
→ Read .claude/plans/active-plan.md

# 3. Presentar resumen
"🔵 Issues in progress: VAL-12 (QueryBuilder), VAL-16 (Narrador)
 📋 Next up: VAL-14 (Quality Gate polish)
 📝 Plan activo: Terminar VAL-12 e2e tests
 ¿Retomamos VAL-12 o cambiamos de foco?"

# 4. Si no hay plan activo → preguntar qué hacer
# 5. Si hay plan activo → retomar desde último checkpoint

/project:end-session — Paso a paso

End

Lo que hace el agente al cerrar

# 1. Generar resumen de sesión
→ Qué se hizo, decisiones tomadas, deviaciones, qué sigue

# 2. Actualizar Linear
→ Linear:save_comment en issues trabajados (resumen de progreso)
→ Linear:save_issue para mover state si se completó
→ Linear:update_document "Session Log — Dev" (append entrada)

# 3. Actualizar plan local
→ Write .claude/plans/active-plan.md con próximos pasos

# 4. Si hubo decisiones importantes
→ Linear:update_document "Decision Log" (append entrada)

# 5. Git
→ Confirmar que todos los commits tienen Refs: VAL-XX
→ Push si no se hizo
08 — StockIA vs D4C

Qué tomamos, qué cambiamos

StockIA (markdown-only)
.stockia/
  ROADMAP.md        → checkboxes manuales
  SESSION_LOG.md    → append manual
  BACKLOG.md        → lista priorizada
  tasks/
    000-xxx/SPEC.md → contrato de tarea
    000-xxx/LOG.md  → execution log

→ Agente lee/escribe filesystem
→ Sin integración con PM tools
→ Funciona offline, 0 API calls
→ Un solo repo, un solo dev
D4C (Linear-first)
Linear/
  Issues           → tareas con ciclos
  Documents        → session log, decisions
  Projects         → roadmap, milestones
  Comments         → progress updates

.claude/
  agents/           → 6 especialistas D4C
  commands/         → Linear-integrated
  plans/            → solo plan táctico local

→ Agente lee/escribe Linear via MCP
→ Equipo de 3 ve todo en un lugar
→ Git hooks para trazabilidad
→ Multi-repo compatible

Qué adoptamos tal cual

Concepto StockIA En D4C
CLAUDE.md <2KB router ✅ Idéntico. Router con reglas y punteros.
Subagentes especializados (.claude/agents/) ✅ Adaptados al contexto D4C (swarm-architect, agent-engineer, etc.)
Slash commands ejecutables ✅ Pero integrados con Linear MCP en vez de leer markdown.
start-session / end-session ✅ "Git pull/push cognitivo" — pero contra Linear, no contra archivos.
docs/ on-demand ✅ ARCHITECTURE, DEVELOPER_GUIDE, DOMAIN_MODEL en repo.
Haiku para tareas simples ✅ pm-linear, test-writer, infra-ops en Haiku.
active-plan.md local ✅ Plan táctico de sesión en filesystem (rápido, no necesita API).

Qué cambiamos

StockIA D4C Por qué
.stockia/ROADMAP.md Linear Projects + Milestones 3 personas necesitan ver el roadmap. Linear es el lugar compartido.
.stockia/SESSION_LOG.md Linear Doc "Session Log — Dev" Persiste entre repos. Pedro y Lorenzo pueden verlo.
.stockia/BACKLOG.md Linear Issues con state=Backlog Filtros nativos, labels, prioridades. No reinventar.
.stockia/tasks/XXX/SPEC.md Linear Issue description El issue ES la spec. Acceptance criteria como checklist en el issue.
.stockia/tasks/XXX/LOG.md Linear Comments en el issue Cada sesión de trabajo = un comment con progreso.
docs/DECISIONS.md Linear Doc "Decision Log" Compartido entre los 3. El agente appenda via MCP.
09 — Flujo diario

Cómo es una sesión de desarrollo

0. Pick issue en Linear

Elegir (o que el agente sugiera) el próximo issue del cycle. Si es nuevo: /project:plan-task lo crea en Linear.

1. /project:start-session

El agente lee Linear (issues in progress + cycle) + plan local. Presenta el estado y pregunta: "¿Retomamos VAL-12 o cambiamos?"

2. Branch + Code

/project:implement-feature VAL-12 — Crea branch con nombre de Linear, ejecuta Domain→App→Infra. Commits con Refs: VAL-12.

3. Test + Review

/project:run-tests → /project:review-code → swarm-architect valida pipeline, infra-ops valida deploy safety.

4. /project:end-session

Actualiza Linear (comment en issue, estado, session log doc). Push. Plan local para próxima sesión. Done.

10 — Plan-task reimaginado

De folder-per-task a issue-per-task

StockIA creaba una carpeta .stockia/tasks/003-nombre/ con SPEC.md + LOG.md. Nosotros creamos un issue en Linear con toda esa info estructurada.

Linear Issue

Anatomía de un issue bien escrito

# [Title] — lo que se va a hacer

## Objetivo
Qué se busca lograr en 2-3 oraciones.

## Contexto
Background técnico y de negocio.

## Scope
**Incluido:** lo que SÍ entra
**Excluido:** lo que NO entra

## Acceptance Criteria
- [ ] Criterio verificable 1
- [ ] Criterio verificable 2
- [ ] Criterio verificable 3

## Implementation Steps
### Fase 1: [nombre]
- [ ] Paso 1
- [ ] CHECKPOINT: mostrar resultado al human
### Fase 2: [nombre]
- [ ] Paso 2

## Risks
| Riesgo | Mitigación |

## Git
Branch: feature/VAL-XX-nombre-corto
Commits: tipo(scope): desc — Refs: VAL-XX
El issue ES la spec. Los comments son el execution log. Los attachments son los artifacts. No necesitás carpeta aparte. Y todo es visible para Pedro y Lorenzo sin clonar el repo.
11 — Reglas de oro

8 reglas que no se rompen

Las reglas del equipo D4C para trabajar con Claude Code

# Regla Por qué
1 Siempre empezar con /project:start-session Es el "git pull" cognitivo. Sin esto, el agente no sabe dónde estamos.
2 Siempre terminar con /project:end-session Es el "git push" cognitivo. Sin esto, el progreso se pierde en Linear.
3 Todo commit lleva Refs: VAL-XX Trazabilidad código↔tarea. Sin esto, el loop Linear→Git se rompe.
4 Las tareas viven en Linear, no en markdown Única fuente de verdad. Si duplicás, se desincroniza.
5 Un issue = una tarea = un branch No mezclar. Si crece, dividir en sub-issues con dependencias.
6 Domain nunca importa de Infrastructure Regla #1 de hexagonal. Si se viola, el swarm se acopla al infra.
7 Tests antes de commit pytest tests/ -v debe pasar. Sin excepciones.
8 Decisiones importantes → Decision Log en Linear Si no lo registrás, lo vas a re-debatir en 2 semanas.

# Cada uno con: instrucciones paso a paso para el agente
# Los que usan Linear incluyen las llamadas MCP explícitas
